# ¿Qué es la Programación Orientada a Aspectos (AOP)?

> (Aspect-Oriented Programming, AOP)

La Programación Orientada a Aspectos es un paradigma que permite separar las funcionalidades transversales (cross-cutting concerns) del código principal de una aplicación.

Su idea principal es:

**"Separar el código de negocio del código que se repite en muchas partes del sistema."**

En lugar de escribir el mismo código una y otra vez, ese comportamiento se encapsula en un aspecto, que se aplica automáticamente donde sea necesario.

**¿Por qué surgió?**

La Programación Orientada a Objetos resolvió muchos problemas organizando el software en clases y objetos.

_Sin embargo, apareció un nuevo inconveniente._

1. Supongamos que desarrollas un sistema bancario con estas clases:

    ```
    Cuenta

    Cliente

    Transferencia

    Préstamo
    ```

    Todas funcionan correctamente.

2. Ahora el banco solicita registrar (hacer logging) cada operación realizada.

    Sin AOP tendrías que hacer algo como:

    + Transferir() → `Registrar Log` → Realizar transferencia
    + Retirar() → `Registrar Log` → Realizar retiro
    + Depositar() → `Registrar Log` → Realizar depósito

    El código para registrar operaciones se repite en muchos lugares.

    Lo mismo ocurre con:

        + Seguridad.
        + Auditoría.
        + Manejo de errores.
        + Medición de tiempo.
        + Transacciones.
        + Caché.

    Este tipo de funcionalidades se denominan **preocupaciones transversales** _(cross-cutting concerns)._

**Analogía**

Imagina un hospital.

Cada médico realiza una tarea distinta:

+ Cardiología.
+ Pediatría.
+ Cirugía.
+ Dermatología.

Pero todos deben cumplir ciertas reglas:

+ Registrar la consulta.
+ Verificar la identidad del paciente.
+ Lavarse las manos.
+ Registrar la receta

**Esas tareas son comunes a todos.**

+ No tendría sentido que cada médico implementara su propio sistema para hacerlo.
+ La administración del hospital define esas reglas y todos las cumplen automáticamente.
+ **Eso es exactamente lo que hace la AOP.**

### ¿Qué es una preocupación transversal?

Una preocupación transversal es una funcionalidad que afecta a muchas partes del sistema.

Por ejemplo:

+ Login → `Validar permisos`
+ Transferencia → `Validar permisos`
+ Eliminar usuario → `Validar permisos`

La validación de permisos aparece en muchos lugares.

_**Con AOP se escribe una sola vez.**_

### Características principales

1. Separación de responsabilidades

    Cada clase se dedica únicamente a su lógica de negocio.

    ```
    Clase Cuenta
    ↓
    Depositar()

    Retirar()
    ```

    No necesita preocuparse por:

    + Auditoría.
    + Logs.
    + Seguridad.

2. Reutilización

    + El aspecto puede utilizarse en muchas clases.
    + Un único aspecto de auditoría puede funcionar para cientos de métodos.

3. Menor duplicación

    En lugar de escribir:

    ```
    Registrar Log()
    Transferir()
    Registrar Log()
    Retirar()
    Registrar Log()
    Depositar()
    ```

    Se escribe:

    Aspecto Log → Se ejecuta automáticamente

4. Código más limpio

    + Las clases contienen únicamente la lógica del negocio.

### Conceptos fundamentales

1. Aspecto (Aspect) 

+ Es el componente principal de la AOP.
+ Representa una funcionalidad transversal.
    + Logging.
    + Seguridad.
    + Auditoría.
    + Caché.
    + Manejo de excepciones

    Representación: `Aspecto` → `Registrar operaciones`

2. Join Point

+ Un Join Point es un punto específico durante la ejecución del programa donde un aspecto puede intervenir.
    + Antes de ejecutar un método.
    + Después de ejecutar un método.
    + Al lanzar una excepción.

    Representación: Método → Join Point → Aspecto

3. Advice

+ El Advice es el código que ejecuta el aspecto.

+ Puede ejecutarse: 

    + Antes del método.
    + Después del método.
    + Alrededor del método.
    + Cuando ocurre una excepción

+ Tipos principales:

| Tipo            | ¿Cuándo se ejecuta?                     |
| --------------- | --------------------------------------- |
| Before          | Antes del método                        |
| After           | Después del método                      |
| Around          | Antes y después                         |
| After Returning | Solo si el método termina correctamente |
| After Throwing  | Solo si ocurre una excepción            |

4. Pointcut

+ Un Pointcut define en qué métodos o lugares debe aplicarse un aspecto.

+ Ejemplo: Todos los métodos que comiencen por guardar*

+ El aspecto solo actuará allí.

5. Weaving

+ El Weaving es el proceso de integrar los aspectos con el código principal.

+ Puede realizarse:
    + Durante la compilación.
    + Durante la carga del programa.
    + Durante la ejecución.

### ¿Cómo funciona?

Supongamos un método:

+ realizarTransferencia()

Con AOP:

```
Before
↓
Verificar permisos
↓
realizarTransferencia()
↓
Registrar auditoría
↓
After
```

+ La clase nunca llamó explícitamente a esos métodos.

+ El aspecto los añadió automáticamente.

### Ventajas

**Código más limpio**
+ Cada clase tiene una única responsabilidad.

**Menos duplicación**
+ Se escribe una sola vez.

**Fácil mantenimiento**
+ Modificar un aspecto actualiza el comportamiento en todo el sistema.

**Mayor reutilización**
+ Los aspectos pueden reutilizarse en muchas aplicaciones.

### Desventajas

**Mayor complejidad**
+ Hay una capa adicional de abstracción.

**Depuración más difícil**
+ El flujo del programa ya no es completamente visible en el código fuente.

**Curva de aprendizaje**
+ Conceptos como Aspect, Advice o Pointcut requieren tiempo para dominarlos.

### Casos de uso

La AOP es muy útil para funcionalidades que afectan a muchas partes de un sistema:

+ Registro de operaciones (logging).
+ Seguridad y autenticación.
+ Auditoría.
+ Manejo de transacciones.
+ Caché.
+ Medición de rendimiento.
+ Reintentos automáticos ante fallos.
+ Gestión centralizada de excepciones.

### AOP vs Programación Orientada a Objetos

Esta comparación ayuda a entender por qué surgió la AOP.

| POO                                               | AOP                                                           |
| ------------------------------------------------- | ------------------------------------------------------------- |
| Organiza el software en clases y objetos.         | Organiza funcionalidades transversales en aspectos.           |
| Se enfoca en entidades del dominio.               | Se enfoca en comportamientos que afectan a muchas entidades.  |
| Reduce la duplicación entre objetos relacionados. | Reduce la duplicación entre módulos no relacionados.          |
| Las responsabilidades están dentro de las clases. | Las responsabilidades transversales se extraen de las clases. |

**Ejemplo comparativo**

Supongamos que tienes tres métodos:

```
crearUsuario()
eliminarUsuario()
actualizarUsuario()
```

**Sin AOP:** 

Se repite el mismo patrón en todos los metodos

+ Registrar Log → Validar permisos → `crearUsuario()` → Registrar auditoría
+ Registrar Log → Validar permisos → `eliminarUsuario()` → Registrar auditoría
+ Registrar Log → Validar permisos → `actualizarUsuario()` → Registrar auditoría

**Con AOP:**

```
crearUsuario()←-----|                   |←-- Aspecto Seguridad
                    |_____ Aspectos ____|       
eliminarUsuario()←--|                   |←-- Aspecto Logging
                    |                   |
actualizarUsuario()←|                   |←-- Aspecto Auditoría...
```

+ El método solo contiene la lógica del negocio.
+ Los aspectos se ejecutan automáticamente.

### Relación con otros paradigmas

La AOP complementa a otros paradigmas en lugar de sustituirlos:

+ **POO + AOP:** una aplicación Java donde las clases representan entidades y los aspectos gestionan seguridad o auditoría
+ **Programación Reactiva + AOP:** registrar automáticamente cada evento procesado
+ **Programación Dirigida por Eventos + AOP:** validar permisos antes de ejecutar un manejador de eventos
+ **Programación Concurrente + AOP:** medir el tiempo de ejecución de tareas concurrentes sin modificar su código

### Comparación con otros paradigmas transversales

| Paradigma                      | Idea principal                                               |
| ------------------------------ | ------------------------------------------------------------ |
| Concurrente                    | Gestionar varias tareas al mismo tiempo.                     |
| Paralelo                       | Ejecutar tareas simultáneamente en varios núcleos.           |
| Reactivo                       | Reaccionar a flujos continuos de datos y eventos.            |
| Dirigido por Eventos           | Responder a eventos individuales.                            |
| **Orientado a Aspectos (AOP)** | Separar funcionalidades transversales del código de negocio. |

### Ejemplo real: Spring Framework

La AOP es ampliamente utilizada en Spring, uno de los frameworks más populares de Java.

Cuando un desarrollador marca un método con una anotación como:

+ `@Transactional`
+ `@Cacheable`
+ `@Secured`

normalmente no escribe el código que inicia la transacción, gestiona la caché o verifica la seguridad. Spring utiliza AOP para interceptar la llamada al método y añadir automáticamente ese comportamiento antes, después o alrededor de la ejecución.

Por eso, aunque muchos desarrolladores no implementen aspectos directamente con AspectJ, utilizan AOP de forma cotidiana a través de frameworks modernos.

### Conclusión

La Programación Orientada a Aspectos no pretende reemplazar a la Programación Orientada a Objetos, sino resolver un problema que esta no aborda por completo: las preocupaciones transversales (cross-cutting concerns)

+ **POO: "Organiza el sistema en objetos que representan el dominio del problema."**
+ **AOP: "Extrae las responsabilidades que se repiten en muchos objetos y las aplica automáticamente donde correspondan."**

Al encapsular esas responsabilidades en aspectos, la AOP consigue aplicaciones con un código más limpio, modular y fácil de mantener

Es un paradigma especialmente valioso en aplicaciones empresariales de gran tamaño, donde funcionalidades como la seguridad, las transacciones, el registro de operaciones y la auditoría deben aplicarse de forma consistente en todo el sistema.
