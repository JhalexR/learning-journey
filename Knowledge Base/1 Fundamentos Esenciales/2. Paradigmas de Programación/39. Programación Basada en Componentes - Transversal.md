# ¿Qué es la Programación Basada en Componentes?
> Component-Based Programming o CBP

Es un paradigma que consiste en construir aplicaciones mediante componentes independientes, reutilizables y encapsulados, que pueden combinarse para formar sistemas más grandes.

Su idea principal es:

**Construir aplicaciones ensamblando componentes reutilizables, como si fueran piezas de un rompecabezas o bloques de LEGO.**

Cada componente tiene una responsabilidad específica y se comunica con los demás a través de _interfaces_ bien definidas.

**¿Por qué surgió?**

La Programación Orientada a Objetos permitió reutilizar clases y objetos.

Sin embargo, cuando los proyectos crecieron, apareció un problema:

+ Una clase es una unidad de reutilización relativamente pequeña
+ Muchas veces se necesitaban módulos **completos** que pudieran reutilizarse sin conocer su implementación interna.

La solución fue crear componentes.

Un componente puede estar formado por: 

+ Varias clases.
+ Archivos de configuración.
+ Recursos gráficos.
+ Servicios.
+ Interfaces.

_Todo encapsulado como una única unidad reutilizable._

**Una analogía sencilla**

Imagina que quieres construir una computadora.

No fabricas:

```
el procesador,
la memoria RAM,
el disco duro,
la tarjeta gráfica.
```

+ Simplemente compras componentes ya construidos y los ensamblas.
+ Cada componente fue desarrollado por separado.
+ Mientras todos respeten los mismos conectores (interfaces), funcionarán juntos.
+ **Eso es exactamente la Programación Basada en Componentes.**

```
Computadora

├── CPU
├── Memoria RAM
├── Disco SSD
├── Tarjeta gráfica
└── Fuente de poder
```

### ¿Qué es un componente?

Un componente es una unidad de software que:

+ Tiene una función específica.
+ Es reutilizable.
+ Está encapsulada.
+ Puede reemplazarse por otra equivalente.
+ Se comunica mediante interfaces.

Por ejemplo, en una tienda en línea podríamos tener:

```
Sistema

├── Componente Usuarios
├── Componente Pagos
├── Componente Inventario
├── Componente Carrito
└── Componente Facturación
```
_Cada componente puede desarrollarse y mantenerse de forma independiente._

### Características principales

1. Reutilización
+ Un componente puede utilizarse en muchos proyectos.

    Ejemplo :
    
    ```Componente Login```
    
    Puede emplearse en:

    + una tienda en línea,
    + un sistema bancario,
    + una aplicación educativa.

    Sin modificar su código.

2. Encapsulamiento

+ El usuario del componente no necesita conocer cómo está implementado.

    Solo necesita saber:

    + qué hace,
    + qué datos recibe,
    + qué resultados devuelve.

3. Bajo acoplamiento

+ Los componentes deben depender lo menos posible entre sí.

    `Componente A` → `Interfaz` → `Componente B`

+ Si cambia la implementación de B, A no debería verse afectado mientras la interfaz permanezca igual.

4. Alta cohesión

+ Cada componente debe encargarse de una única responsabilidad.

    Ejemplo:

        No debería también:

        - generar reportes,
        - imprimir facturas,
        - procesar pagos.

5. Intercambiabilidad

+ Un componente puede reemplazarse por otro compatible.

    Ejemplo: 

    _**Sistema**_ → `Componente Pago PayPal`

    Puede cambiarse por:

    _**Sistema**_ → `Componente Pago Stripe`

    _Siempre que ambos implementen la misma interfaz._

### Componentes e Interfaces

Las `interfaces` son fundamentales en este paradigma.

Una `interfaz` define cómo interactuar con un componente, sin revelar su implementación.

Por ejemplo:

```
Interfaz Pago
↓
procesarPago()
↓
cancelarPago()
```

No importa si el componente utiliza PayPal, Stripe o cualquier otro proveedor.

Todos ofrecen la misma `interfaz`.

### ¿Cómo funciona?

Supongamos un sistema de biblioteca.

```
Sistema

├── Gestión de Libros
├── Gestión de Usuarios
├── Préstamos
├── Multas
└── Reportes
```

_Cada módulo es un componente._

Cuando un usuario solicita un préstamo:

```
Usuario
↓
Componente Préstamos
↓
Componente Libros
↓
Componente Usuarios
```

_Cada componente realiza únicamente su función._

### Ventajas

**Reutilización**
+ Un componente puede utilizarse en muchos sistemas.

**Mantenimiento sencillo**
+ Si hay un error en el componente de pagos, solo se modifica ese componente.

**Desarrollo en paralelo**
+ Diferentes equipos pueden trabajar en componentes distintos.

    + `Equipo A` → `Usuarios`
    + `Equipo B` → `Pagos`
    + `Equipo C` → `Inventario`

+ Todos trabajan simultáneamente.

**Escalabilidad**
+ Es más fácil agregar nuevos componentes sin modificar los existentes.

**Facilidad para realizar pruebas**
+ Cada componente puede probarse de forma aislada.

### Desventajas

**Diseño inicial más complejo**
+ Es necesario definir correctamente las interfaces y las responsabilidades de cada componente.

**Gestión de dependencias**
+ Un componente puede depender de otros, por lo que es importante controlar esas relaciones.

**Sobrecarga de comunicación**
+ Si los componentes interactúan demasiado entre sí, pueden aparecer pérdidas de rendimiento.

### Ejemplos reales

**React**

En React, una página se construye mediante componentes.

```
Página

├── Navbar
├── Menú
├── Perfil
├── Productos
└── Footer
```

Cada componente puede reutilizarse en distintas páginas.

**Angular**

También organiza la interfaz mediante componentes independientes.

**Unity**

Un personaje puede estar formado por varios componentes:

```
Jugador

├── Movimiento
├── Física
├── Sonido
├── Animación
└── Inventario
```

Cada componente añade una funcionalidad específica.

**JavaBeans**

Un JavaBean es un componente reutilizable dentro del ecosistema Java.

**.NET**

Blazor utiliza componentes para construir interfaces web.

### Programación Basada en Componentes vs. Programación Orientada a Objetos

> _Esta es una comparación muy importante._

| POO                              | Basada en Componentes                    |
| -------------------------------- | ---------------------------------------- |
| Unidad principal: Clase u Objeto | Unidad principal: Componente             |
| Reutilización de clases          | Reutilización de módulos completos       |
| Encapsula datos y comportamiento | Encapsula funcionalidades completas      |
| Comunicación mediante objetos    | Comunicación mediante interfaces         |
| Escala bien a nivel de clases    | Escala mejor a nivel de sistemas grandes |

**Podemos decir que un componente normalmente está formado por muchas clases.**

### Relación con otros paradigmas

La Programación Basada en Componentes suele combinarse con otros enfoques:

+ **POO + Componentes:** `Spring`, `JavaBeans`, `Blazor` y muchas aplicaciones empresariales.
+ **Reactiva + Componentes:** `React` y `Angular` reaccionan a cambios en los datos actualizando componentes.
+ **Dirigida por Eventos + Componentes:** un botón (componente) genera eventos que otros componentes manejan.
+ **AOP + Componentes:** aspectos como seguridad o auditoría pueden aplicarse a todos los componentes.
+ **Concurrente + Componentes:** distintos componentes pueden ejecutar tareas en paralelo o de forma concurrente.

Casos de uso

La Programación Basada en Componentes se utiliza ampliamente en:

+ Interfaces gráficas (GUI).
+ Aplicaciones web modernas.
+ Aplicaciones móviles.
+ Videojuegos.
+ Sistemas empresariales.
+ Frameworks de desarrollo.
+ Arquitecturas de microservicios (donde cada servicio puede verse como un componente de mayor tamaño).

### Comparación con otros paradigmas transversales

| Paradigma                 | Idea principal                                            |
| ------------------------- | --------------------------------------------------------- |
| Concurrente               | Gestionar múltiples tareas.                               |
| Paralelo                  | Ejecutar tareas simultáneamente.                          |
| Reactivo                  | Reaccionar a flujos de datos y eventos.                   |
| Dirigido por Eventos      | Responder a eventos individuales.                         |
| AOP                       | Separar funcionalidades transversales.                    |
| **Basado en Componentes** | Construir sistemas ensamblando componentes reutilizables. |

### ¿En qué se diferencia de los microservicios?

Es común confundir estos conceptos.

+ **Programación Basada en Componentes:** `los componentes` suelen ejecutarse dentro de una misma aplicación y comparten el mismo proceso.
+ **Arquitectura de Microservicios:** cada `servicio` es una aplicación independiente que se comunica con las demás mediante la red (`HTTP`, `gRPC`, `mensajería`, etc.).

_Podemos decir que un `microservicio` es una unidad de despliegue, mientras que un `componente` es una unidad de construcción del software._

### Conclusión

La Programación Basada en Componentes propone construir aplicaciones como si se ensamblaran piezas de LEGO

cada componente tiene: 
+ una responsabilidad clara
+ ofrece una interfaz bien definida
+ puede reutilizarse en distintos sistemas

Facilita el desarrollo de aplicaciones grandes, donde varios equipos pueden trabajar en paralelo, reutilizar componentes existentes y mantener el sistema de forma más sencilla.

Frameworks modernos como React, Angular, Blazor y motores como Unity demuestran que el desarrollo basado en componentes se ha convertido en una de las formas más comunes de construir software.

Una forma sencilla de recordar este paradigma es:

+ **POO:** "Organiza el software en `objetos` que colaboran entre sí."
+ **Programación Basada en Componentes:** "Organiza el software en `módulos completos`, _reutilizables_ e _intercambiables_, que se ensamblan para construir aplicaciones más grandes."
