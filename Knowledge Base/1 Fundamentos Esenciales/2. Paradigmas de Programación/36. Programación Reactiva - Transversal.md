# ¿Qué es la Programación Reactiva?

La programación reactiva es un paradigma que se basa en la propagación de eventos y cambios de estado.

En lugar de que un programa pregunte constantemente si algo ha cambiado, reacciona automáticamente cuando ocurre un evento.

Su idea principal es:

+ "Cuando ocurre un evento, el sistema responde automáticamente ejecutando las acciones correspondientes."

**¿Por qué surgió?**

Las aplicaciones modernas reciben continuamente eventos:

+ Un usuario hace clic
+ Llega un mensaje
+ Se recibe información desde Internet
+ Cambia un sensor
+ Se actualiza una base de datos
+ Se recibe una notificación

En los programas tradicionales era común revisar periódicamente si había novedades, lo que podía ser ineficiente.

La programación reactiva propone otro enfoque:
+ Los datos notifican sus cambios y los componentes interesados reaccionan automáticamente.

### Conceptos fundamentales

1. Evento (Event)

Un evento es cualquier suceso que ocurre durante la ejecución del programa.

+ Pulsar un botón.
+ Mover el mouse.
+ Llegar un correo electrónico.
+ Recibir datos desde un servidor.
+ Finalizar una descarga.
```
Evento -> Ejecutar acción
```
2. Flujo de eventos (Event Stream)

Los eventos normalmente no llegan aislados.

```
Evento 1
↓
Evento 2
↓
Evento 3
↓
Evento 4
↓
Evento 5
```
Este flujo continuo se denomina stream (flujo).

3. Observador (Observer)

+ El observador es quien espera los eventos.
+ Cuando aparece uno nuevo, reacciona.
```
Flujo
↓
Observador
↓
Procesar evento
```

4. Observable

Es el objeto que produce los eventos.
```
Sensor
↓
Temperatura
↓
Observadores
```
Cada vez que cambia la temperatura, el sensor informa a todos los observadores.

**Ejemplo cotidiano**

Sin programación reactiva:
```
Cada segundo
↓
Consultar servidor
↓
¿Cambió la temperatura?
```

Con programación reactiva:
```
Servidor
↓
Nueva temperatura
↓
Aplicación actualiza pantalla
```

Solo se trabaja cuando ocurre un cambio.

### Características principales

1. Basada en eventos

+ Todo gira alrededor de eventos.
+ No existe un flujo rígido de instrucciones.
+ Los eventos determinan qué ocurre y cuándo.

2. Flujo continuo de datos

+ Los datos pueden llegar de forma permanente.
+ Cada nuevo valor genera una reacción.

    Sensor -> 20°C -> 21°C -> 22°C -> 21°C

3. Programación asíncrona

+ La mayoría de los sistemas reactivos son asíncronos.
+ Mientras esperan nuevos eventos pueden seguir realizando otras tareas.

4. Composición de operaciones

+ Los eventos pueden transformarse mediante operaciones encadenadas.
+ Cada paso recibe el resultado del anterior.

    Datos -> Filtrar -> Ordenar -> Transformar -> Mostrar
    
5. Baja latencia

+ Las respuestas ocurren inmediatamente después del evento.
+ No es necesario esperar una consulta periódica.

### Ventajas

**Interfaces muy dinámicas**

+ Ideal para aplicaciones modernas.
    + Ejemplo: Mientras escribes en Google aparecen sugerencias automáticamente.

**Menor consumo de recursos**
+ No es necesario preguntar continuamente si hubo cambios.

**Escalable**
+ Puede manejar miles o millones de eventos.

**Muy útil para sistemas en tiempo real**

+ Bolsa de valores.
+ Redes sociales.
+ Videojuegos en línea.
+ Sistemas IoT.

### Desventajas

**Curva de aprendizaje**
+ Pensar en eventos y flujos es diferente a pensar en algoritmos secuenciales.

**Depuración**
+ Seguir el recorrido de un evento puede ser más complejo que seguir un programa lineal.

**Mayor complejidad conceptual**

Hay conceptos nuevos como:

+ Observable.
+ Observer.
+ Stream.
+ Subscription.
+ Operadores.

### Componentes principales

**Observable**

Produce los eventos:

```
Observable
↓
Evento 1
↓
Evento 2
↓
Evento 3
```

**Observer**

Consume los eventos:
```
Observable
↓
Observer
```

**Suscripción (Subscription)**

Conecta un observador con un observable.
```
Observable
↓
Suscripción
↓
Observer
```

**Operadores**

Permiten modificar el flujo de eventos.

+ filtrar
+ transformar
+ agrupar
+ combinar
+ limitar

### Ejemplo de la vida real

**Spotify**

Mientras escuchas música:

+ llegan nuevas canciones
+ cambian los controles
+ aparece la letra
+ se actualiza el progreso
+ cambia la portada

Cada cambio es un evento.

La interfaz reacciona automáticamente.

### Casos de uso

La programación reactiva es especialmente útil cuando los datos llegan de manera continua o impredecible:

+ Aplicaciones web modernas.
+ Interfaces gráficas (GUI).
+ Aplicaciones móviles.
+ Chats en tiempo real.
+ Redes sociales.
+ Sistemas de monitoreo.
+ Internet de las Cosas (IoT).
+ Streaming de audio y video.
+ Sistemas financieros con cotizaciones en tiempo real.
+ Microservicios que intercambian eventos.

### Relación con otros paradigmas

La programación reactiva no sustituye a los paradigmas tradicionales; los complementa.

Por ejemplo:

+ **POO + Reactiva:** una aplicación Android en Kotlin donde las pantallas reaccionan a cambios en los datos.
+ **Funcional + Reactiva:** usar operadores como map, filter y reduce sobre flujos de eventos.
+ **Concurrente + Reactiva:** un servidor que atiende múltiples solicitudes mientras procesa eventos de forma asíncrona.
+ **Paralela + Reactiva:** procesar varios flujos de datos simultáneamente en distintos núcleos cuando la carga lo requiere.

### Programación Reactiva vs Programación Dirigida por Eventos

_**Estos conceptos están muy relacionados, pero no son idénticos.**_

| Programación Dirigida por Eventos                   | Programación Reactiva                                                      |
| --------------------------------------------------- | -------------------------------------------------------------------------- |
| Se centra en responder a eventos individuales.      | Se centra en procesar flujos continuos de eventos.                         |
| Cada evento suele manejarse de forma independiente. | Los eventos pueden filtrarse, transformarse, combinarse y encadenarse.     |
| Es común en interfaces gráficas.                    | Es común en sistemas distribuidos y procesamiento de datos en tiempo real. |

**La programación reactiva puede verse como una evolución de la programación dirigida por eventos, ya que añade herramientas para tratar los eventos como secuencias de datos.**

### Programación Reactiva vs. Programación Concurrente

_**Esta comparación suele generar muchas dudas.**_

| Programación Concurrente                                       | Programación Reactiva                                            |
| -------------------------------------------------------------- | ---------------------------------------------------------------- |
| Gestiona varias tareas que avanzan de forma coordinada.        | Gestiona flujos de eventos y reacciones a cambios.               |
| Se enfoca en la ejecución de tareas.                           | Se enfoca en el flujo de datos y eventos.                        |
| Utiliza hilos, procesos o tareas asíncronas.                   | Utiliza observables, observadores y operadores sobre flujos.     |
| El objetivo es mejorar la capacidad de realizar varias tareas. | El objetivo es responder de forma eficiente a cambios y eventos. |

**Un sistema reactivo puede ser concurrente, pero la programación reactiva no depende obligatoriamente de tener varios hilos.**

### Conclusión

La programación reactiva es un paradigma transversal que propone un cambio de perspectiva: en lugar de escribir programas que controlan constantemente el estado del sistema, se construyen programas que reaccionan automáticamente cuando ocurre un evento o cambia un dato

Este enfoque es especialmente valioso en el desarrollo de aplicaciones modernas, donde las interfaces de usuario, los servicios en la nube, los dispositivos IoT y los sistemas distribuidos generan flujos continuos de información. Gracias a su integración natural con la programación asíncrona y a su capacidad para procesar eventos de forma eficiente, la programación reactiva se ha convertido en una herramienta fundamental para construir software escalable, receptivo y fácil de mantener.
