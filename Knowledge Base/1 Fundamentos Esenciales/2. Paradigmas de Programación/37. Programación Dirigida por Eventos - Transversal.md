# ¿Qué es la Programación Dirigida por Eventos?
> (Event-Driven Programming, EDP)

La Programación Dirigida por Eventos es un paradigma en el que el flujo de ejecución del programa está determinado por la ocurrencia de eventos.

En lugar de ejecutar instrucciones de forma estrictamente secuencial, el programa permanece "esperando" hasta que ocurre un evento y, cuando sucede, ejecuta el código asociado.

Su idea principal es:
+ Cuando ocurre un evento, ejecuta el manejador (handler) correspondiente.

> Es uno de los paradigmas más importantes en el desarrollo de software moderno. De hecho, prácticamente todas las aplicaciones con interfaz gráfica, aplicaciones web, videojuegos y aplicaciones móviles utilizan este paradigma.

### ¿Qué es un evento?

Un evento es cualquier acción o suceso que puede ser detectado por un programa.

+ Un clic del mouse
+ Pulsar una tecla
+ Mover el cursor
+ Recibir un correo
+ Llegar un mensaje por la red
+ Finalizar una descarga
+ Cambiar el valor de un sensor
+ Expirar un temporizador

**¿Por qué surgió este paradigma?**

Los primeros programas eran secuenciales, Pero cuando aparecieron las interfaces gráficas surgió un problema.

Un programa debía responder continuamente a acciones del usuario.

+ mover ventanas
+ escribir texto
+ hacer clic
+ cambiar de pantalla

Era imposible conocer el orden en que ocurrirían esas acciones.

_La solución fue construir programas que permanecieran esperando eventos._

**¿Cómo funciona?**

Todo programa dirigido por eventos tiene un ciclo principal llamado _Event Loop._

Su funcionamiento es: 

```
Esperar evento -> ¿Ocurrió? -> Sí -> Ejecutar manejador -> Volver a esperar
```

El programa nunca deja de esperar nuevos eventos.

### Componentes principales

1. Evento (Event)

+ Es el suceso que ocurre.

    + Click
    + KeyPress
    + MouseMove
    + Scroll
    + Timer
    + Mensaje recibido

2. Fuente del evento (Event Source)

+ Es quien genera el evento.

+ `Boton` -> `Click`

+ `Teclado` -> `KeyPress`

3. Listener (Escuchador)

+ Es el componente que permanece atento a un tipo de evento.

+ `Botón` -> `Listener` -> `Espera Click`

4. Event Handler (Manejador)

+ Es la función que se ejecuta cuando ocurre el evento.

+ `Click` -> `guardarArchivo()`

### Características principales

1. Basado en eventos

+ El flujo del programa depende de eventos externos.
+ No existe un orden fijo.

2. Event Loop

+ Siempre existe un ciclo esperando eventos.
+ `Esperar` -> `Evento` -> `Procesar`->`Esperar`

3. Manejadores independientes

+ Cada evento tiene su propio código.
+ `Click botón Guardar` -> `guardar()`
+ `Click botón Salir` -> `cerrarPrograma()`

4. Desacoplamiento

+ El botón no necesita saber qué hace el código.
+ Solo genera el evento.
+ El manejador decide cómo responder.

5. Interactividad

+ Es ideal para aplicaciones donde el usuario participa constantemente.

### Ventajas

**Interfaces muy dinámicas**
+ Perfecto para aplicaciones gráficas.

**Fácil agregar funcionalidades** 
+ Solo se crea un nuevo manejador.

**Código modular**
+ Cada evento tiene su propia lógica.

**Alta interacción con el usuario**

Ideal para:

+ aplicaciones móviles
+ videojuegos
+ páginas web
+ sistemas de escritorio

### Desventajas

**Flujo menos evidente**

+ No siempre es fácil seguir el orden de ejecución.
+ Depende de qué eventos ocurran.

**Muchos eventos**
+ Aplicaciones grandes pueden tener cientos de manejadores.

**Depuración más difícil**
+ Un error puede aparecer únicamente cuando ocurre cierta combinación de eventos.

### Ejemplo real

Una aplicación bancaria.

+ ``Usuario pulsa Transferir`` -> `Evento` -> `Validar datos` -> `Consultar saldo` -> `Realizar transferencia` -> `Mostrar mensaje`

Todo comienza por un único evento.

### Event Loop

_**El Event Loop es el corazón de este paradigma.**_

Representación: 

```
Esperar evento Mientras programa esté abierto -> ¿Llegó? -> Sí -> Ejecutar handler
    ↑_______________________________________________________________________↓
```

Así funcionan:

+ Windows
+ Android
+ iOS
+ JavaScript
+ Node.js
+ Navegadores web

### Programación Dirigida por Eventos vs. Programación Reactiva

**Esta es una de las dudas más frecuentes.**

| Programación Dirigida por Eventos              | Programación Reactiva                                                            |
| ---------------------------------------------- | -------------------------------------------------------------------------------- |
| Se centra en responder a eventos individuales. | Se centra en procesar flujos continuos de datos y eventos.                       |
| Cada evento dispara un manejador específico.   | Los eventos forman secuencias (*streams*) que pueden transformarse y combinarse. |
| El foco está en el evento.                     | El foco está en el flujo de datos.                                               |
| Muy común en interfaces gráficas.              | Muy común en sistemas distribuidos, tiempo real y procesamiento de datos.        |

Ejemplo de Diferencias:

+ **Cuando haces clic en un botón que guarda un archivo, eso es programación dirigida por eventos.**

+ **Si una aplicación recibe continuamente datos de sensores, los filtra, los transforma y actualiza un panel en tiempo real, estamos ante un enfoque reactivo**

### Comparación con otros paradigmas transversales

| Paradigma            | Idea principal                                          | Ejemplo                                                   |
| -------------------- | ------------------------------------------------------- | --------------------------------------------------------- |
| Concurrente          | Gestionar múltiples tareas.                             | Un servidor atendiendo varios clientes.                   |
| Paralelo             | Ejecutar tareas simultáneamente para ganar rendimiento. | Procesar imágenes en varios núcleos.                      |
| Reactivo             | Reaccionar a flujos de datos y cambios de estado.       | Actualizar un panel con datos de sensores en tiempo real. |
| Dirigido por Eventos | Responder a eventos individuales cuando ocurren.        | Ejecutar una acción al hacer clic en un botón.            |

### Conclusión

La Programación Dirigida por Eventos es uno de los paradigmas transversales más utilizados porque permite construir aplicaciones interactivas, donde el flujo del programa no está determinado por una secuencia fija de instrucciones, sino por las acciones del usuario o por sucesos del entorno.

+ Es la base de prácticamente todas las interfaces gráficas modernas y de gran parte del desarrollo web y móvil
+ sirve como fundamento para comprender paradigmas más avanzados como la Programación Reactiva

Una forma sencilla de recordar diferencias entre EDP y Reactiva:

+ **Programación Dirigida por Eventos:** "Ocurre un evento → ejecuto una acción."
+ **Programación Reactiva:** "Llega un flujo continuo de eventos o datos → observo, transformo y reacciono a ese flujo."

Esta diferencia hace que ambos paradigmas sean complementarios y, en muchas aplicaciones modernas, trabajen juntos.
