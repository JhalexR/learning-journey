# ¿Qué es la Programación Concurrente?

Se considera un paradigma transversal porque puede combinarse con otros paradigmas como la programación imperativa, orientada a objetos o funcional.

En la práctica, cuando se habla de paradigmas modernos, es completamente válido incluir la programación concurrente.

La programación concurrente es un paradigma en el que *varias tareas pueden estar en progreso al mismo tiempo.*

La idea principal es:

+ **Dividir un problema en varias tareas independientes que puedan ejecutarse de forma concurrente, coordinándose entre sí cuando sea necesario.**

```
No significa necesariamente que todas las tareas se ejecuten exactamente al mismo tiempo (paralelismo)

Las tareas avanzan de manera intercalada o simultánea según los recursos disponibles.
```

Antes los computadores tenian un solo procesador, Todo era secuencial, pero los computadores comenzaron a incorporar: 

+ varios núcleos (multicore)
+ múltiples procesadores
+ servidores distribuidos
+ aplicaciones que realizan muchas tareas simultáneamente

Entonces apareció un nuevo problema:

+ ¿Cómo hacer que varias tareas trabajen "al mismo tiempo" sin interferirse?

> De ahí nació la programación concurrente.

**ejemplo sencillo**

Con concurrencia

```
Mesa 1 → tomar pedido
↓
Enviar cocina
↓
Mientras cocina...
↓
Tomar pedido mesa 2
↓
Cobrar mesa 3
↓
Llevar bebidas mesa 4
↓
Regresar por comida mesa 1
```
+ Ahora el mesero puede organizar mejor su tiempo.
+ Varias tareas avanzan "al mismo tiempo".

> Eso es concurrencia.

### ¿Qué significa "tarea"?

Una tarea puede ser, dependiendo del lenguaje:

+ un hilo (thread)
+ un proceso
+ una rutina
+ una corrutina (coroutine)
+ un actor
+ una tarea asíncrona

### Objetivo principal

Aprovechar mejor los recursos del computador.

Mientras un programa espera:

+ una respuesta de Internet
+ un archivo
+ una base de datos

puede seguir haciendo otras cosas.

Supongamos una aplicación de música.

Mientras reproduce una canción también:

+ descarga la siguiente
+ muestra la portada
+ responde a los botones
+ sincroniza la lista de reproducción

Todo parece ocurrir simultáneamente.

### Características

**Varias tareas activas**
+ No existe un único flujo de ejecución.

**Coordinación**
+ Las tareas deben comunicarse.

**Sincronización**

A veces varias tareas necesitan el mismo dato.

Ejemplo:

Dos cajeros quieren modificar el saldo de una cuenta bancaria.

Si ambos escriben al mismo tiempo puede ocurrir un error.

Por eso existen mecanismos como:

+ mutex
+ locks
+ semáforos
+ monitores
+ variables atómicas

**Comunicación**

Las tareas pueden compartir información mediante:

+ memoria compartida
+ paso de mensajes
+ canales
+ colas

**Ventajas**

+ **Mejor aprovechamiento del hardware** -> Especialmente en CPUs multinúcleo.
+ **Mayor capacidad de respuesta** -> La interfaz no se bloquea.
+ **Mayor rendimiento** -> Muchos servidores atienden miles de usuarios concurrentemente.
+ **Escalabilidad** 
    + Ideal para:
        + servidores web
        + videojuegos
        + aplicaciones móviles
        + sistemas financieros
        + inteligencia artificial
        + sistemas distribuidos

**Desventajas**

+ **Es mucho más difícil de programar**
    + Ya no existe un único flujo.
    + Hay varios ejecutándose al mismo tiempo.

+ **Condiciones de carrera (Race Conditions)**
    + Dos tareas modifican la misma información.
    
        + `Saldo` = 100
        + `Hilo A` resta 50
        + `Hilo B` resta 30
    
    + Dependiendo del orden de ejecución el resultado puede ser incorrecto.

+ **Deadlocks**
    + Dos tareas quedan esperando indefinidamente.
        + Proceso A espera al recurso B
        + Proceso B espera al recurso A
    + Ninguno puede continuar.

+ **Depuración complicada**
    + Los errores pueden aparecer solo algunas veces, dependiendo del orden en que se ejecuten las tareas.

### Concurrencia vs Paralelismo

**Es uno de los conceptos más importantes.**

**Concurrencia**

+ Hay varias tareas en progreso.
+ Puede existir un solo procesador.
+ La CPU alterna rápidamente entre ellas.
+ *Parece que todas avanzan al mismo tiempo.*

```
CPU -> A -> B -> A -> C -> B -> A
```

**Paralelismo**

+ Varias tareas realmente se ejecutan al mismo tiempo.
+ Cada núcleo trabaja simultáneamente.

```
CPU 1 → Tarea A
CPU 2 → Tarea B
CPU 3 → Tarea C
```
**Muchos lenguajes modernos incorporan herramientas para este paradigma:**

| Lenguaje   | Soporte para concurrencia             |
| ---------- | ------------------------------------- |
| Java       | Threads, Executors, CompletableFuture |
| C#         | async/await, Tasks, Threads           |
| Go         | Goroutines y Channels                 |
| Rust       | Threads, async/await, Tokio           |
| Kotlin     | Coroutines                            |
| Python     | threading, multiprocessing, asyncio   |
| JavaScript | Promises, async/await, Web Workers    |
| Erlang     | Actores (Actors)                      |
| Elixir     | Actores sobre la máquina virtual BEAM |

### Relación con otros paradigmas

A diferencia de la programación procedural o funcional, la programación concurrente no reemplaza a los demás paradigmas, sino que los complementa. Un mismo programa puede ser:

+ **Orientado a Objetos** + *Concurrente* (por ejemplo, una aplicación Java con múltiples hilos).
+ **Funcional** + *Concurrente* (como muchas aplicaciones en Elixir o Haskell).
+ **Imperativo** + *Concurrente* (por ejemplo, un servidor escrito en Go).
+ **Procedural** + *Concurrente* (como un programa en C utilizando hilos POSIX).

_Por eso suele decirse que la concurrencia es un **paradigma transversal:**_

+ paradigmas tradicional: describen cómo se _estructura y escribe el código._
+ paradigma transversal: describe cómo se _organizan y ejecutan las tareas._

## ¿Que es la programación con hilos?

Es una parte fundamental del paradigma de programación concurrente.

En lugar de un paradigma independiente, **es una técnica de concurrencia** que permite que un programa realice varias tareas al mismo tiempo

### ¿Qué es un hilo?

+ Un hilo (o thread) es la unidad más pequeña de procesamiento que puede realizar una tarea.

+ Mientras un proceso representa una aplicación completa, un hilo representa una tarea específica dentro de esa aplicación.

### ¿Qué resuelve la programación con hilos?

+ **Velocidad:** Hace que los programas sean más rápidos al repartir el trabajo.
+ **Fluidez:** Evita que una aplicación se congele mientras realiza tareas pesadas (como descargar un archivo).

Ejemplo:

```
Proceso: Navegador Web

├── Hilo 1 → Dibujar la interfaz
├── Hilo 2 → Descargar imágenes
├── Hilo 3 → Ejecutar JavaScript
└── Hilo 4 → Reproducir videos
```
Todos los hilos comparten:

+ La misma memoria.
+ Las mismas variables globales.
+ Los mismos archivos abiertos.

Por eso son rápidos para comunicarse, pero también pueden generar problemas si dos hilos modifican el mismo dato al mismo tiempo.

**Diferencias entre Proceso e Hilo**

| Aspecto       | Proceso   | Hilo                                   |
| ------------- | --------- | -------------------------------------- |
| Independiente | Sí        | No                                     |
| Memoria       | Propia    | Compartida con otros hilos del proceso |
| Comunicación  | Más lenta | Muy rápida                             |
| Creación      | Costosa   | Más ligera                             |
| Aislamiento   | Alto      | Bajo                                   |

### Sincronización de tareas

**La regla de oro: La sincronización**

Cuando varias tareas comparten recursos, es necesario coordinarlas.

La sincronización consiste en establecer reglas para que varias tareas trabajen sin interferirse.

+ Se necesitan reglas estrictas para compartir información (memoria)
+ Si dos hilos intentan usar el mismo recurso a la vez, pueden causar errores
+ Para evitar esto, se usa la **sincronización**
+ Es como poner un semáforo que da el turno a cada hilo

### Condiciones de Carrera (Race Conditions)

Una condición de carrera ocurre cuando el resultado depende del orden en que se ejecutan las tareas.

```
contador = 0

Dos hilos hacen:
    contador = contador + 1

Lo esperado sería:

    contador = 2

Pero podría suceder:

Hilo A lee 0
Hilo B lee 0
Hilo A escribe 1
Hilo B escribe 1

Resultado:

contador = 1 En lugar de 2.
```

El problema no es el código, sino el momento en que cada hilo ejecuta sus instrucciones.

### Deadlocks y Livelocks

**Deadlock (Interbloqueo)**

Sucede cuando dos o más tareas quedan esperando indefinidamente recursos que posee la otra.

Ejemplo:

```
Proceso A

    Tiene Recurso 1
    Espera Recurso 2

Proceso B

    Tiene Recurso 2
    Espera Recurso 1

A espera a B
↓
B espera a A
↓
Nadie continúa
```
El programa queda bloqueado.

**Livelock**

En un livelock, las tareas no están bloqueadas, pero tampoco avanzan porque reaccionan continuamente entre sí.

**Analogía**

Dos personas intentan pasar por un pasillo estrecho.

+ Ambas se mueven a la izquierda.
+ Ambas cambian a la derecha.
+ Ambas vuelven a la izquierda.

Nunca chocan, pero tampoco avanzan.

En programación ocurre algo similar: los procesos siguen ejecutándose, pero ninguno progresa.

**Diferencia**

| Deadlock                      | Livelock                         |
| ----------------------------- | -------------------------------- |
| Los procesos quedan detenidos | Los procesos siguen ejecutándose |
| No hacen nada                 | Trabajan constantemente          |
| No existe progreso            | Tampoco existe progreso          |


### Mecanismos de sincronización

Para evitar problemas como las condiciones de carrera, existen varias herramientas.

1. Mutex (Mutual Exclusion)

    Un mutex permite que solo un hilo acceda a un recurso a la vez.
    
    Hilo A
    ↓
    Bloquea el recurso
    ↓
    Trabaja
    ↓
    Libera el recurso
    ↓
    Hilo B puede entrar
    
    Es como un baño con llave: mientras una persona lo usa, las demás esperan.


2. Semáforos

    Un semáforo controla cuántos hilos pueden acceder simultáneamente a un recurso.

    Ejemplo:

    Una impresora con capacidad para atender tres trabajos al mismo tiempo.

    
    Capacidad = 3
    Hilo A → entra
    Hilo B → entra
    Hilo C → entra
    Hilo D → espera
    

3. Monitores

    Un monitor combina:

        un recurso compartido
        un mecanismo de exclusión mutua
        y operaciones para esperar o notificar

    Muchos lenguajes, como Java, implementan monitores de forma nativa con la palabra clave synchronized.

    Su objetivo es facilitar la sincronización sin que el programador tenga que gestionar manualmente los bloqueos.


4. Barreras (Barriers)

    Una barrera obliga a que todos los hilos lleguen a un punto antes de continuar.

    Hilo A
    ↓
    Llega

    Hilo B
    ↓
    Llega

    Hilo C
    ↓
    Llega

    Solo cuando todos han llegado, continúan.

    Se utilizan mucho en simulaciones y algoritmos paralelos.

### Modelos de concurrencia

Los modelos de concurrencia definen cómo cooperan las tareas entre sí.

**1. Memoria compartida**

    Todos los hilos acceden a las mismas variables.

    Hilo A
    ↓
    Memoria Compartida
    ↑
    Hilo B

    Es muy eficiente, pero requiere sincronización para evitar conflictos.

    Java, C++, C#.

**2. Paso de mensajes (Message Passing)**

    Las tareas no comparten memoria.

    Se comunican enviándose mensajes.
    Proceso A
    ↓
    Mensaje
    ↓
    Proceso B

    Es más seguro porque evita que dos tareas modifiquen directamente los mismos datos.

    MPI o sistemas distribuidos.

**3. Modelo Actor**

    Cada actor:

        Tiene su propio estado
        Procesa un mensaje a la vez
        Y se comunica únicamente mediante mensajes

        Actor A
        ↓
        Mensaje
        ↓
        Actor B
        ↓
        Mensaje
        ↓
        Actor C

        Esto reduce muchos problemas de sincronización.

        Lenguajes como Erlang y Elixir utilizan este modelo de forma intensiva.

**4. CSP (Communicating Sequential Processes)**

    En este modelo, los procesos son independientes y se comunican mediante canales.

    Proceso A
    ↓
    Canal
    ↓
    Proceso B

    El lenguaje Go adopta esta filosofía con sus goroutines y channels.

**Comparación**

| Modelo             | Comunicación      | Memoria compartida |
| ------------------ | ----------------- | ------------------ |
| Memoria compartida | Variables comunes | Sí                 |
| Paso de mensajes   | Mensajes          | No                 |
| Actor              | Mensajes          | No                 |
| CSP                | Canales           | No                 |

### Programación asíncrona

La programación asíncrona es una técnica que permite iniciar una operación larga sin detener el resto del programa

No necesariamente implica varios hilos; en muchos casos, un único hilo puede gestionar múltiples operaciones esperando eventos.

Ejmeplo Código asíncrono

```
Iniciar descarga
↓
Seguir trabajando
↓
Responder al usuario
↓
Actualizar pantalla
↓
Cuando termina la descarga
↓
Procesar el archivo
```

El programa sigue siendo útil mientras espera.

**async / await**

Permite escribir código asíncrono con una sintaxis muy similar a la secuencial.

Conceptualmente:

```
Iniciar operación
↓
Esperar sin bloquear
↓
Continuar cuando termine
```

> Está presente en lenguajes como C#, Python, JavaScript, Kotlin y Rust.

**Promesas (Promises)**

Una promesa representa un resultado que estará disponible en el futuro.

```
Solicitar datos al servidor
↓
Promise
↓
Datos disponibles
↓
Procesar respuesta
```

> Es muy utilizada en JavaScript.

**Futures**

Un Future es similar a una promesa: representa el resultado de una tarea que aún no ha finalizado, pero que podrá consultarse cuando termine.

Es común en Java, Scala y otros lenguajes de la JVM.

## Resumen general

| Concepto               | ¿Qué representa?                               | Ejemplo cotidiano                                                  |
| ---------------------- | ---------------------------------------------- | ------------------------------------------------------------------ |
| Proceso                | Programa en ejecución                          | Un navegador web abierto                                           |
| Hilo (Thread)          | Tarea dentro de un proceso                     | Descargar imágenes mientras navegas                                |
| Sincronización         | Coordinar tareas                               | Turnarse para usar una impresora                                   |
| Condición de carrera   | Acceso simultáneo sin control                  | Dos personas escribiendo en la misma hoja a la vez                 |
| Deadlock               | Espera mutua indefinida                        | Dos personas bloqueando una puerta, esperando que la otra se mueva |
| Livelock               | Movimiento constante sin avanzar               | Dos personas esquivándose continuamente en un pasillo              |
| Mutex                  | Exclusión mutua                                | La llave de un baño: solo entra una persona                        |
| Semáforo               | Limitar accesos simultáneos                    | Un estacionamiento con plazas limitadas                            |
| Monitor                | Recurso protegido con sincronización integrada | Una oficina con recepcionista que organiza el acceso               |
| Barrera                | Esperar a que todos lleguen                    | Una carrera que empieza cuando todos están en la línea de salida   |
| Memoria compartida     | Compartir datos directamente                   | Varias personas escribiendo en una misma pizarra                   |
| Paso de mensajes       | Intercambiar información mediante mensajes     | Enviar cartas entre oficinas                                       |
| Modelo Actor           | Entidades independientes que procesan mensajes | Empleados con buzones individuales                                 |
| CSP                    | Comunicación mediante canales                  | Una cinta transportadora entre estaciones                          |
| Programación asíncrona | Continuar trabajando mientras una tarea espera | Poner una lavadora y aprovechar para cocinar mientras termina      |

### Conclusión

Todos estos conceptos están relacionados entre sí y forman la base de la programación concurrente moderna. Una buena forma de entender su relación es pensar en tres niveles:

**1. Unidades de ejecución:** procesos e hilos, que realizan el trabajo.

**2. Coordinación y seguridad:** sincronización, mutex, semáforos, monitores y barreras, que permiten compartir recursos sin errores como condiciones de carrera o deadlocks.

**3. Modelos y estilos de diseño:** memoria compartida, paso de mensajes, actores, CSP y programación asíncrona, que definen distintas formas de organizar la comunicación y colaboración entre tareas concurrentes.
