# Paradigmas transversales

Existen algunos paradigmas de programación que algunos analistas consideran paradigmas por si mismos, pero aun la mayoria los consideran como un modelo de ejecución dentro de otros paradigmas.

A estos paradigmas se les suele llamar "Paradigmas transversales".

La mayoría de los lenguajes modernos son multiparadigma, permitiendo mezclar estas ideas para resolver problemas, por esto es que es importante entender un poco de estos paradigmas transversales, que estan tomando mas importancia en la industria.

## ¿Qué es la Programación Procedural?

la programación procedural suele generar mucha confusión porque muchas personas creen que es un paradigma completamente distinto de la programación imperativa. En realidad, la programación procedural es una especialización o un estilo dentro del paradigma imperativo.

La programación procedural (Procedural Programming) es un paradigma que organiza un programa en procedimientos o funciones, donde cada procedimiento realiza una tarea específica.

Mientras que la programación imperativa solo indica una secuencia de instrucciones, la programación procedural divide esa secuencia en módulos reutilizables.

Su idea principal es:

+ "Divide un problema grande en pequeños procedimientos que puedan reutilizarse."

**Una analogía sencilla**

Imagina que debes preparar una pizza.

**Programación Imperativa**

Harías todo seguido:

```
1. Preparar masa
2. Agregar salsa
3. Agregar queso
4. Agregar ingredientes
5. Hornear
6. Servir
```
> Todo ocurre en un único bloque.

**Programación Procedural**

Ahora divides el trabajo:

```
inicio

hacerMasa()

hacerSalsa()

agregarIngredientes()

hornear()

servir()

fin
```

+ El programa principal simplemente llama esos procedimientos.
+ Cada procedimiento puede reutilizarse.

**¿Qué es un procedimiento?**

Un procedimiento es un conjunto de instrucciones agrupadas bajo un nombre.

### Características principales

**Modularidad**

+ El programa se divide en pequeñas partes. 
+ En lugar de escribir 500 líneas de código se dividen las instrucciones en procedimientos(modulos).
+ Cada procedimiento(modulo) hace una sola cosa.

**Reutilización**

+ Un procedimiento puede llamarse muchas veces.

Ejemplo:
```
sumar(5,4)
sumar(20,15)
sumar(100,8)
```
+ No se vuelve a escribir el mismo código.

**Flujo secuencial**

```
Inicio

↓

Procedimiento A

↓

Procedimiento B

↓

Procedimiento C

↓

Fin
```

**Uso de variables**

Las variables cambian durante la ejecución.

```
contador = 0

contador = contador + 1

contador = contador + 1
```
Existe un estado mutable.

**Paso de parámetros**

Los procedimientos reciben información.

Ejemplo:

```
calcularArea(base, altura)
```
> Dependiendo de los parámetros producen resultados diferentes.

**Ventajas**

+ **Código organizado** -> Cada procedimiento tiene una responsabilidad.
+ **Fácil mantenimiento** -> Si cambia una operación solo se modifica un procedimiento.
+ **Reutilización** -> No se duplica código.
+ **Fácil de aprender** -> Por eso suele enseñarse primero en las universidades.
+ **Buena legibilidad** -> Los nombres de los procedimientos describen qué hacen.

**Desventajas**

+ **Variables globales** -> En muchos programas procedurales clásicos existían variables globales que podían modificarse desde cualquier procedimiento. Esto dificultaba el mantenimiento.
+ **Poco escalable** -> Cuando el programa crece mucho aparecen cientos de procedimientos -> Puede resultar difícil entender qué procedimiento llama a cuál.
+ **Modela mal objetos reales** -> Si estás construyendo un videojuego o un sistema bancario, pensar solo en procedimientos puede ser menos natural que modelar entidades como `Jugador` `Cuenta` `Cliente`

**Relación con la Programación Imperativa**

La relación puede verse como una jerarquía:

```
Paradigmas de programación

│

├── Imperativo
│      │
│      ├── Procedural
│      └── Orientado a Objetos
│
├── Funcional
│
└── Declarativo
```

La programación procedural toma la base imperativa de  instrucciones y cambios de estado y añade una organización del código mediante procedimientos o funciones.

Es decir:

+ Todo programa procedural es imperativo.
+ No todo programa imperativo está bien estructurado de forma procedural.

**Comparación: Imperativa vs. Procedural** 

| Aspecto              | Programación Imperativa             | Programación Procedural               |
| -------------------- | ----------------------------------- | ------------------------------------- |
| Enfoque              | Secuencia de instrucciones          | Procedimientos o funciones            |
| Organización         | Puede ser un único bloque de código | Código dividido en módulos            |
| Reutilización        | Limitada                            | Alta mediante funciones               |
| Mantenimiento        | Más difícil en programas grandes    | Más sencillo gracias a la modularidad |
| Nivel de abstracción | Bajo                                | Un poco más alto al encapsular tareas |

**Conclusión**

La programación procedural fue un paso muy importante en la evolución del desarrollo de software. Permitió pasar de programas monolíticos a programas organizados en funciones reutilizables, sentando las bases de prácticas modernas como la modularidad, la separación de responsabilidades y la reutilización de código.

Aunque hoy en día la Programación Orientada a Objetos domina muchos proyectos empresariales y la Programación Funcional ha ganado relevancia en el procesamiento de datos y la concurrencia, la programación procedural sigue siendo fundamental. De hecho, muchos programas escritos en lenguajes modernos como Python, C, Go o incluso Java utilizan un estilo predominantemente procedural cuando el problema no requiere el uso de clases y objetos.

En otras palabras, más que un paradigma "nuevo", la programación procedural es una forma de organizar la programación imperativa de manera más clara, mantenible y reutilizable. Comprenderla te dará una base sólida para entender cómo evolucionaron los paradigmas posteriores y por qué muchos conceptos actuales, como las funciones y los módulos, siguen siendo esenciales en el desarrollo de software
